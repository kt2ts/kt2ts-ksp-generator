# 0.0.16 and 0.0.17

Two bugs surfaced while running kt2ts on a multi-module backend whose root
annotated classes (sealed `Command` / `Query` / `QueryResponse`) reach a lot
of types declared in a shared `common` library — and whose data model
combines generics, type aliases, and several types mapped to hand-written
TS files via `kt-to-ts-mappings.json`.

Both bugs left the generated TS *almost* correct: the file compiled
internally but referenced symbols that were neither emitted nor imported.
End result: the front-end TypeScript build broke even though `kspKotlin`
succeeded.

## 0.0.16 — mapped types nested inside generics were not imported

### Symptom

```kotlin
data class GetAirtableUpdatesQueryResponse(
    val providersDiffDto: DiffWithInserts<ProviderInformations>,
    …
) : QueryResponse()
```

With `lite.common.domain.ProviderInformations` mapped to
`domain/provider-offer.ts`, the generated file came out as:

```ts
export interface GetAirtableUpdatesQueryResponse {
  …
  providersDiffDto: DiffWithInserts<ProviderInformations>;  // unimported
}
```

`ProviderInformations` was used as a type argument but no
`import { ProviderInformations } from '@lite-eco/backoffice/domain/provider-offer'`
was emitted. Same story for any mapped type used inside `List<…>`,
`Set<…>`, `Map<…>`, or any user generic.

### Root cause

`Kt2TsSymbolProcessor` builds the import list by walking each parsed
class's properties and asking `ClassMapper.mapProperty(propertyType, …)`
for a `ClassMapping`. `mapProperty` looks at the **outer** type only:

- Outer type is `DiffWithInserts<ProviderInformations>` →
  `DiffWithInserts` is not in the mappings → `mapProperty` returns `null`.
- The inner `ProviderInformations` is never visited.

The other source of imports (`Parsed.dependencies` collected during
parsing) also doesn't help here. `ClassParser.mapDependencies` keeps only
the **unmapped** dependencies in the set (it filters mapped ones because
those won't need to be re-parsed). So a mapped type used solely as a type
argument falls into neither source.

### Fix

A new helper that walks the type tree and returns every applicable
mapping — outer plus every nested type argument, recursively:

```kotlin
fun collectMappedTypes(
    t: KSTypeReference,
    mappings: Map<String, String>,
    mapClassMapping: ClassMapping?,
): List<ClassMapping> {
    val self = listOfNotNull(mapProperty(t, mappings, mapClassMapping))
    val args = t.element?.typeArguments
        ?.mapNotNull { it.type }
        ?.flatMap { collectMappedTypes(it, mappings, mapClassMapping) }
        ?: emptyList()
    return self + args
}
```

`Kt2TsSymbolProcessor.dependenciesImportsMapped` now uses
`flatMap { collectMappedTypes(propertyType, …) }` instead of
`mapNotNull { mapProperty(propertyType, …) }`.

The downstream import deduplication (`groupBy { it.tsFile }`) already
collapses repeats, so emitting "too many" mappings is harmless.

## 0.0.17 — type aliases were ignored by the mapping lookup

### Symptom

```kotlin
// backoffice/domain/diff.kt
typealias ObjectDiffMap = Map<FieldName, Any?>
```

The mapping `"lite.backoffice.domain.ObjectDiffMap": "domain/diff.ts"`
was correctly placed in the JSON. The generator nonetheless emitted:

```ts
export interface DiffDto {
  …
  inserts: ObjectDiffMap[];  // unimported, undefined
}
```

The mapping JSON entry had no effect.

### Root cause

`ClassMapper.mapProperty` started with:

```kotlin
val d = t.resolve().declaration as? KSClassDeclaration ?: return null
val qualifiedName = d.qualifiedName?.asString()
if (qualifiedName in mappings.keys) { … }
```

A Kotlin `typealias` resolves to a `KSTypeAlias`, not a
`KSClassDeclaration`. The cast on the first line returns `null` and the
function bails out **before ever checking the mappings table**.

### Fix

Check the mappings on the raw declaration first, and cast to
`KSClassDeclaration` only when falling back to the built-in
(`Boolean` / `Int` / `List` / `Set` / `Map` / `Pair` / `Any`) handling:

```kotlin
val rawDecl = t.resolve().declaration
val qualifiedName = rawDecl.qualifiedName?.asString()
if (qualifiedName != null && qualifiedName in mappings.keys) {
    return ClassMapping(rawDecl.simpleName.asString(), mappings.getValue(qualifiedName))
}
val d = rawDecl as? KSClassDeclaration ?: return null
return when (qualifiedName) { … }
```

Combined with 0.0.16, this also lets the import collector pick up the
mapping for `ObjectDiffMap` and emit the corresponding TS `import`.

## Notes for the future rewrite

A few things worth carrying over when kt2ts is rewritten on top of a
proper Kotlin parser (instead of KSP):

- "What's a dependency to descend into" and "what's a symbol to import"
  are two different sets. Mapped types belong to the second but not the
  first; type parameters belong to neither. The current code blurs them
  in `Parsed.dependencies` and pays for it with bolt-on helpers like
  `collectMappedTypes`. A clean rewrite should separate the two from
  the start.
- The mapping lookup should be the **first** thing the property/type
  resolver does, before any reasoning about the declaration kind
  (class, alias, parameter…). Otherwise alias-like constructs keep
  slipping through.
- The cross-module gap is still real: a referenced type whose declaring
  file is outside the module being compiled is silently dropped unless
  it appears in the mappings JSON. The new parser should at minimum
  warn (and probably fail) when it emits a referenced-but-not-imported
  identifier.
